Chess Opening Trainer — Feature Ideas


---


1. Opening DNA Fingerprint

Description: Analyze a user's game history (via PGN upload or Lichess/Chess.com sync) to identify which openings they gravitate toward and which they consistently misplay. Generate a visual "DNA strand" showing their opening repertoire strengths and gaps. User value: Players immediately see where they're losing theory points without manually reviewing dozens of games. Complexity: Medium

---


2. Spaced Repetition Drill Mode

Description: Apply an Anki-style SRS algorithm to opening lines, surfacing positions the user has missed or hesitated on at optimal review intervals. Each position is a flashcard — the user plays the move, gets instant feedback, and the card is scheduled accordingly. User value: Moves get committed to long-term memory with minimal wasted repetition. Complexity: Medium

---


3. Grandmaster Ghost Mode

Description: Play through an opening alongside a "ghost" overlay of how a famous grandmaster (Fischer, Kasparov, Magnus) historically played it — same position, their actual game choices shown as faint arrows. Users can diverge and see where they part ways. User value: Provides aspirational context and makes rote drilling feel like a mentorship session. Complexity: High

---


4. Opening Story Mode

Description: Each opening comes with a short narrative — the historical moment it was born, the rivalry that shaped it, the tournament that popularized it. Moves are revealed as chapters in the story, so learning the Sicilian feels like reading a thriller. User value: Transforms dry memorization into memorable storytelling, dramatically improving retention. Complexity: Low–Medium

---


5. Trap Detector & Trap Setter

Description: Highlight the sharpest traps in any opening line — positions where a natural-looking opponent move leads to a quick loss. Users can practice springing traps on a bot or studying how to sidestep them when playing Black. User value: Gives club-level players immediately actionable "weapons" they can use next weekend. Complexity: Medium

---


6. Opponent Style Simulator

Description: Instead of a static engine, pair the opening trainer with bots that mimic common amateur playing styles — the "e4 e5 beginner," the "London System loyalist," the "hyper-aggressive Sicilian attacker." Users practice openings against realistic human patterns, not perfect computer play. User value: Bridges the gap between knowing theory and applying it against real opponents. Complexity: High

---


7. Opening Report Card

Description: After each practice session, generate a one-page report card grading accuracy, speed of recall, and depth reached in each line. Include a trend chart showing improvement over the past 30 days and a single "focus line" recommendation for next session. User value: Creates a satisfying feedback loop and a clear next step, reducing decision fatigue. Complexity: Low

---

---


Fatal Flaw: Eliminating #6 (Opponent Style Simulator)


This idea has a near-fatal execution risk. The promise — bots that *feel* like authentic human opponents — requires either ML models trained on enormous amateur game corpora, or heuristics convincing enough that users suspend disbelief. Neither is achievable at "High" complexity; this is an order of magnitude harder.


When style simulators fall short (and they almost always do), they don't just fail quietly — they actively undermine trust in the whole product. A user who notices the "aggressive Sicilian bot" makes computer-like defensive moves will discount every other feature. Cut it.


Grandmaster Ghost Mode (#3) is borderline. Game database licensing is a real legal surface, and rendering ghost-arrow overlays cleanly on a board is deceptively tricky UX. It also skews toward intermediate+ players while features 2, 4, 5, and 7 serve a broader range. Defer to a later phase, not eliminate.

---


Ranking (Remaining 6 Ideas)


Tier 1 — Ship First (Core Loop)

1. Spaced Repetition Drill Mode — This *is* the product. Without it you're a static opening reference, not a trainer. Chessable proved the market; the opportunity is to do it better.

2. Opening Report Card — Minimal effort, maximum retention. It completes the feedback loop for every session. No excuse not to have this at launch.


Tier 2 — Ship Second (Differentiation)

3. Trap Detector & Trap Setter — Best acquisition feature on the list. "Learn traps to win next weekend" is a concrete, testable promise. Drives word-of-mouth among club players.

4. Opening DNA Fingerprint — Personalization creates stickiness. The "DNA strand" visual is gimmicky; call it a Repertoire Map and it's stronger. Lichess API integration is well-documented and free.


Tier 3 — Later (Engagement/Retention Depth)

5. Opening Story Mode — Genuinely differentiated. Competitors don't do this. But it's only as good as the writing — underfund content and it dies on arrival.

6. Grandmaster Ghost Mode — High ceiling, high risk. Worth revisiting once there's a user base to justify the licensing cost.


---


Convergence Decision


The ideas are not quite ready to converge. Two gaps need addressing before committing to a roadmap:


1. No defined target user. Features 2 and 7 serve beginners; features 1 and 5 serve club players (~1200–1800 Elo); feature 3 serves intermediate+ players. These require different content investments and go-to-market angles. The list needs a primary persona before sequencing makes sense.


2. No clear MVP boundary. Tier 1 above is the right core, but the brief doesn't address the minimum viable content library (how many openings? lines per opening?) needed to make SRS useful at all. Without that, the engineering plan is underspecified.


One more round focused on: (a) locking a primary user persona, (b) defining the opening content scope, and (c) validating whether Trap Detector or DNA Fingerprint is the stronger acquisition hook for that persona.

One refinement round needed to resolve persona and opening library depth.

Creative PM

Chess Opening Trainer — Round 3: Persona Lock & Scope Definition


---


Addressing the Two Convergence Gaps First


Primary Persona: The Aspiring Club Player (800–1600 Elo)


Profile: Plays 3–5 games per week online (Lichess/Chess.com), occasionally OTB. Has a vague "opening repertoire" cobbled together from YouTube videos. Loses games in the opening not from missing deep theory but from forgetting move 6, walking into a well-known trap, or entering a pawn structure they don't understand. Studies in 20–30 minute sessions, usually on mobile.
Why this persona over beginners or advanced players:
  • Large addressable pool (this is the majority of active online chess players)
  • High pain: they know enough to be frustrated by opening mistakes, not enough to self-study efficiently
  • Trap Detector and SRS deliver *immediate* wins — this persona sees value in session one
  • Chessable's weakness here is UX complexity; there's a clear opening

MVP Content Scope


10 openings × 4 main lines × ~8 positions per line = ~320 training positions.

The 10 openings (by popularity at 800–1600 Elo): Italian, Ruy López, Sicilian (Najdorf + Dragon), French, Caro-Kann, London, King's Indian, Queen's Gambit, Scandinavian, Pirc. This is achievable with a small content team and gives White *and* Black coverage. Each line goes 8–10 moves deep — enough to get past the opening without encyclopedic depth.


Acquisition Hook: Trap Detector Wins


For the 800–1600 persona, Trap Detector is the stronger hook. "Learn 3 traps you can use this weekend" is a concrete, social, immediately rewarding promise. DNA/Repertoire Map requires uploading game history — friction before first value. Lead with Trap Detector in marketing, build SRS as the product engine underneath.


---


Refined Feature Set (Persona-Aligned)


---


1. Spaced Repetition Drill Mode *(Retained — Core Loop)*

Description: Anki-style SRS applied to the 320 MVP positions. User plays the move; the card is scheduled based on response speed and accuracy. Positions missed more than twice surface a hint layer — a highlighted square, not the full answer — before revealing. Mobile-first: a full session takes under 15 minutes. User value: The opening becomes automatic. Club players stop spending mental energy on move 6 and start thinking about the middlegame. Complexity: Medium

---


2. Opening Report Card *(Retained — Feedback Loop)*

Description: End-of-session summary: accuracy %, average recall speed, deepest line reached, and one recommended focus line for next session. After 30 days, a trend sparkline shows streak and improvement curve. Shareable as an image. User value: Closes the feedback loop every session. The single "focus next" recommendation eliminates decision fatigue. Complexity: Low

---


3. Trap Detector & Trap Setter *(Retained — Acquisition Hook)*

Description: Every opening line in the MVP library is tagged with its sharpest trap positions. "Trap Mode" is a standalone flow: pick an opening, play through three positions where a common amateur mistake leads to a quick loss, and practice springing the same trap on a puzzle bot. Tagged with approximate Elo range where the trap lands. User value: Immediate, concrete weapon for the next game. "I learned the Stafford Gambit trap in 10 minutes and won with it tonight" is word-of-mouth gold. Complexity: Medium

---


4. Repertoire Map *(Refined from DNA Fingerprint)*

Description: After 10+ drill sessions (no upload required — built from in-app behavior), generate a visual map of the user's opening coverage: lines drilled, accuracy by line, and obvious gaps. Clicking a gap drops the user directly into that line's SRS queue. No gimmicky "DNA" framing — this is a navigator, not a novelty. User value: Replaces the anxiety of "what should I study?" with a clear, personalized answer grounded in actual practice data. Complexity: Medium *(reduced: no external API dependency in v1 — built from first-party drill data)*

---


5. *(New)* Blunder Benchmark

Description: For each opening line, surface the three moves where players at the user's self-reported Elo most commonly go wrong — pulled from aggregated anonymized session data or, at launch, curated from published game databases. Presented as a "96% of players at 1200 play Nc6 here — here's why it loses." Users drill specifically on these high-error positions first. User value: Targets study time at the exact mistakes this persona is statistically likely to make, not the mistakes grandmasters make. Complexity: Low–Medium *(curation-heavy at launch; data-driven post-scale)*

---


6. *(New)* Pre-Game Warm-Up

Description: A 5-minute mode triggered before a game: user inputs "I'm playing White against a French player" (or auto-detected from Lichess account), and the app surfaces a rapid-fire 10-position drill of the most critical French lines. No new material — pure activation of what they've already learned. User value: Bridges the gap between practice and performance. Primes muscle memory immediately before it's needed, not days earlier. Complexity: Low *(built on existing SRS data + simple filtering logic)*

---


Deferred (Not Cut)


  • Opening Story Mode — Differentiated and worth building, but content quality is the risk. Commission 3 pilot stories (Sicilian, King's Indian, London) before greenlighting the full library.
  • Grandmaster Ghost Mode — Revisit at 10k users. Licensing cost becomes justifiable; user base to absorb the UX learning curve exists.

---


Convergence Recommendation


The list is ready to converge. Persona is locked (aspiring club player, 800–1600), content scope is defined (10 openings, ~320 positions), and the feature sequence has a clear logic:


Launch: SRS Drill Mode + Opening Report Card + Trap Detector Month 2: Blunder Benchmark + Pre-Game Warm-Up Month 4: Repertoire Map Phase 2: Story Mode pilots → Ghost Mode

The only remaining open question is build-vs-license for the chess board UI (Lichess's open-source board vs. a custom renderer) — an engineering decision, not a product one.

Critical PM

Critical PM Evaluation: Chess Opening Trainer Round 3


---


Persona & Scope Assessment


The persona lock is solid. 800–1600 is the right target — large pool, clear pain, monetizable frustration. The 320-position MVP scope is credible but I want to flag a hidden risk: content quality gates the entire product. SRS on bad or incomplete content is worse than no SRS. Before committing to 10 openings, confirm who owns curation and what the quality bar is.


---


Feature Scoring


| Feature | Feasibility | User Impact | Effort | Notes |

|---|---|---|---|---|

| SRS Drill Mode | 8 | 9 | 6 | Proven mechanic, but mobile board UX is the hard part |

| Opening Report Card | 9 | 7 | 3 | Easy win — mostly data aggregation + display |

| Trap Detector & Setter | 7 | 9 | 5 | High virality ceiling; puzzle bot scoping is the risk |

| Repertoire Map | 7 | 8 | 5 | Good that external API dependency is removed |

| Blunder Benchmark | 6 | 8 | 7 | Data is the bottleneck — "curation-heavy at launch" is a real cost |

| Pre-Game Warm-Up | 9 | 8 | 2 | Highest ROI feature on the list — underrated |


---


Fatal Flaw Check


No fatal flaws, but two conditional flags:
Blunder Benchmark — watch the data problem. "Aggregated anonymized session data" doesn't exist at launch. You're launching with curated data, which means a content team decision disguised as a product feature. The value prop ("96% of 1200-Elo players make this mistake") requires *your* player data to be credible, not published game databases filtered by Elo. At launch this is essentially hand-labeled content. That's fine — just name it honestly in the roadmap so engineering doesn't over-build the data pipeline prematurely.
Trap Setter puzzle bot — scope it tightly. "Practice springing the trap on a puzzle bot" is doing a lot of work in one sentence. A full puzzle bot is a non-trivial build. If this means "play the winning continuation from the trap position against a static response tree," that's low effort. If it means "dynamic bot that plays realistic defense," that's a different product. Pin this down before Month 1 ends.

---


Ranking (Launch Priority Order)


1. Pre-Game Warm-Up — Highest effort-to-value ratio on the board. Costs almost nothing to build; directly bridges the practice-performance gap that is the product's core promise. Should be in launch, not Month 2.

2. SRS Drill Mode — Non-negotiable core. The whole product fails without this working well on mobile.

3. Trap Detector — Acquisition engine. Correct call to lead marketing here.

4. Opening Report Card — Retention mechanism. Low effort, keeps users coming back.

5. Blunder Benchmark — Strong concept, but sequencing it to Month 2 is correct given data dependency.

6. Repertoire Map — Month 4 is right. Requires real usage data to be meaningful, not just functional.


---


One Sequencing Challenge


Pre-Game Warm-Up is listed as Month 2, but it's the feature most likely to trigger daily active use. Club players play games several times a week — if Pre-Game Warm-Up becomes a habit, you get 3–5 sessions per week instead of 1–2. That retention multiplier matters for Month 1 numbers. Strongly recommend pulling it into launch.


---


Convergence Decision


The product direction is sound. Persona is locked, scope is defined, and the feature logic is coherent. The roadmap is buildable. The two items above (Blunder Benchmark data source transparency, Trap Setter scope pin) are execution risks, not strategic ones.


One open question to resolve before sprint planning: who owns content? The 320-position library, trap tagging, blunder curation — this is the product's moat and its biggest non-engineering cost. If that ownership isn't named, the roadmap will slip.

---


Tech Stack:
  • Frontend: React 18 + TypeScript + Vite, `react-chessboard` + `chess.js`, Zustand (session state), TanStack Query (server state), Tailwind CSS, Recharts (sparklines), Framer Motion
  • Backend: Fastify 4 + TypeScript, Drizzle ORM on PostgreSQL 16, Redis (refresh tokens + rate-limit counters), Zod schemas shared between packages
  • Testing: Vitest (unit/integration on both api and web), Playwright (E2E)
  • Monorepo: pnpm workspaces, `packages/shared`, `packages/api`, `packages/web`

Schema highlights: 8 tables — `users`, `openings`, `positions` (tree), `cards` (one per user×position), `sessions`, `session_items`, `trap_encounters`. Refresh tokens live in Redis (key `rt:<userId>:<tokenId>`, TTL 30d).
SRS algorithm: SM-2 variant in a pure `srs.ts` module (no I/O). Learning steps: 10 min → 1 day; review intervals from 3d → 8d → 21d → ease-factor-scaled. Ease starts at 2.5, ±0.15/−0.20 on correct/hint/incorrect, floored at 1.3.
6 implementation phases (18 days), with Phase 4 (Report Card) and Phase 5 (Trap Mode) independent and parallelisable after Phase 3.
`run_tests.sh`: spins up a test postgres (port 5433) + redis (port 6380) via `docker-compose.test.yml`, runs migrations + seed, then `pnpm --filter shared/api/web test --run`, then Playwright E2E, then tears down.

  • SRS tests are pure — all 21 tests exercise `nextInterval`/`buildSession` with no I/O; exact interval arithmetic is asserted (10min = 1/144 day, graduation to 3d, 8×2.5≈21d)
  • Accessibility is first-class — every E2E spec calls `checkA11y(page)`, plus dedicated `accessibility.spec.ts` covering aria-live, focus trapping, and color-not-sole-indicator
  • JWT attacks tested — `alg:none` and tampered-payload tokens verified to return 401
  • `run_tests.sh` uses a `trap EXIT` to always tear down Docker, accumulates per-suite exit codes, and exits non-zero if any suite fails
