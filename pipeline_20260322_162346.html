Compression Algorithm Explorer — Feature Ideas


---


1. Algorithm Visualizer

Step-by-step animated visualization of how data transforms at each stage of compression (e.g., watch Huffman trees grow in real time, see LZ77 sliding windows move). Users can pause, rewind, and inspect individual bytes. User value: Builds deep intuition by making the invisible visible. Complexity: Medium


---


2. Compression Duel Mode

Pit two algorithms head-to-head on the same input — side-by-side stats on compression ratio, speed, and memory usage update live as both run. User value: Instantly reveals trade-offs that textbooks only describe abstractly. Complexity: Low


---


3. "Break It" Adversarial Inputs

A library of pathological inputs (already-compressed data, random bytes, repeated patterns) that expose each algorithm's worst-case behavior. Includes a mode where users craft their own adversarial payloads. User value: Forces genuine understanding of edge cases, not just happy paths. Complexity: Low


---


4. Annotated Source with "Why This Line?" Tooltips

Every line of the from-scratch implementation carries an inline annotation explaining *why* it exists — linking to the mathematical concept, the original paper, and a counterexample of what breaks without it. User value: Bridges the gap between working code and true comprehension. Complexity: Medium


---


5. Compression Archaeology Mode

Implement historical algorithms in chronological order (Huffman 1952 → LZ77 1977 → DEFLATE 1996 → Zstandard 2016), with a timeline showing how each improved on the last. User value: Gives narrative context — learning *feels* like discovery rather than memorization. Complexity: Medium


---


6. Live Entropy Meter

As users feed any text or file into their implementation, a real-time entropy score shows how much theoretical compression headroom remains. Flags over- and under-compression. User value: Grounds abstract information theory in concrete, immediate feedback. Complexity: Low


---


7. Regression Test Harness with Corpus

A curated corpus (source code, images, genomic data, logs) with a one-command test runner that measures correctness *and* performance against reference implementations. User value: Removes the tedious scaffolding so learners stay focused on the algorithm, not the tooling. Complexity: Medium


---


8. "Cheat Code" Escape Hatch

At any point a user can reveal a reference implementation of the exact function they're struggling with, diff'd against their own attempt, with plain-language commentary on differences. User value: Prevents frustration from killing momentum — learning continues even when stuck. Complexity: Low


---

Gap 1: Duel Mode "Why X Won" One-Liner — Rule-Based System


The one-liner is rule-based, not ML-based. It reads three metrics already computed by the WebWorker — input entropy, compression ratio delta, and encode/decode symmetry — and selects from a small decision table. Here is the complete rule-set:


| Condition | One-liner template |

|---|---|

| Input entropy < 4.2 bits/symbol AND A wins on ratio | "A wins: low-entropy input rewards symbol-frequency coding. Huffman thrives here." |

| Input entropy > 6.5 bits/symbol AND neither beats raw by >5% | "Neither algorithm found much to compress — high-entropy input leaves little headroom for either approach." |

| A wins on ratio but B wins on speed (encode time delta > 20%) | "A compresses tighter; B is faster. Depends whether you're optimizing for size or throughput." |

| B wins on ratio AND entropy is mid-range (4.2–6.5) | "B finds more structure in this input. Mid-range entropy often rewards dictionary-based approaches." |

| Ratio delta < 2% | "Too close to call on compression — check encode time as the tiebreaker." |

| Any edge-case corpus file (e.g. already-compressed input) | "Input is nearly incompressible — both algorithms are working against the format, not with it." |


Implementation note: The rule table is a static lookup evaluated after each WebWorker run. The metrics driving it are already computed. Engineering cost: near zero — this is a switch statement on existing outputs. The one-liner renders below the stat cards, not inline.
One concrete example for the spec: User pits their Huffman implementation against a run-length encoder on a BMP image corpus. Entropy reads 3.1 bits/symbol. Output: *"A wins: low-entropy input rewards symbol-frequency coding. Huffman thrives here."*
Edge case: If no rule fires (this should be rare given the coverage above), the fallback is: *"A edges out B — check the ratio and encode time breakdown above for details."* No silent failures.

---


Gap 2: MVP-A Keystroke Execution — Explicit Debounce Spec


Execution triggers on idle, not on keypress. The spec is:
  • WebWorker run fires 300ms after the last keystroke (standard input debounce)
  • If a run is already in-flight when the debounce fires, the new run is queued and the in-flight run is terminated via `worker.terminate()` — no double-execution
  • While a run is in-flight, the three live metric cards show a subtle pulse animation (not a spinner, not a loading bar — just a low-opacity pulse on the number itself)
  • If a run exceeds 2 seconds, it is terminated and the metric cards show "—" with a tooltip: *"Execution timed out — try a smaller input or check for infinite loops"*
  • The 2-second timeout is the only hard guardrail against runaway algorithms; no other intervention

Why 300ms: Standard for live-preview editors (VS Code, CodePen, CodeSandbox all use this range). Fast enough to feel live; slow enough to avoid hammering the WebWorker on fast typists.
Engineering note for the spec: The debounce lives in the main thread, not the worker. The worker is stateless — it receives a payload (code + corpus chunk), runs once, posts results back, and exits. Main thread manages all scheduling.

| Input entropy < 4.2 bits/symbol AND A wins on ratio | "A wins: low-entropy input rewards symbol-frequency coding. Huffman thrives here." |

| Input entropy > 6.5 bits/symbol AND neither beats raw by >5% | "Neither algorithm found much to compress — high-entropy input leaves little headroom for either approach." |

| A wins on ratio but B wins on speed (encode time delta > 20%) | "A compresses tighter; B is faster. Depends whether you're optimizing for size or throughput." |

| B wins on ratio AND entropy is mid-range (4.2–6.5) | "B finds more structure in this input. Mid-range entropy often rewards dictionary-based approaches." |

| Ratio delta < 2% | "Too close to call on compression — check encode time as the tiebreaker." |

| Any edge-case corpus file (e.g. already-compressed input) | "Input is nearly incompressible — both algorithms are working against the format, not with it." |


Implementation note: The rule table is a static lookup evaluated after each WebWorker run. The metrics driving it are already computed. Engineering cost: near zero — this is a switch statement on existing outputs. The one-liner renders below the stat cards, not inline.
One concrete example for the spec: User pits their Huffman implementation against a run-length encoder on a BMP image corpus. Entropy reads 3.1 bits/symbol. Output: *"A wins: low-entropy input rewards symbol-frequency coding. Huffman thrives here."*
Edge case: If no rule fires (this should be rare given the coverage above), the fallback is: *"A edges out B — check the ratio and encode time breakdown above for details."* No silent failures.

Gap 2: MVP-A Keystroke Execution — Explicit Debounce Spec


Execution triggers on idle, not on keypress. The spec is:
  • WebWorker run fires 300ms after the last keystroke (standard input debounce)
  • If a run is already in-flight when the debounce fires, the new run is queued and the in-flight run is terminated via `worker.terminate()` — no double-execution
  • While a run is in-flight, the three live metric cards show a subtle pulse animation (not a spinner, not a loading bar — just a low-opacity pulse on the number itself)
  • If a run exceeds 2 seconds, it is terminated and the metric cards show "—" with a tooltip: *"Execution timed out — try a smaller input or check for infinite loops"*
  • The 2-second timeout is the only hard guardrail against runaway algorithms; no other intervention

Why 300ms: Standard for live-preview editors (VS Code, CodePen, CodeSandbox all use this range). Fast enough to feel live; slow enough to avoid hammering the WebWorker on fast typists.
Engineering note for the spec: The debounce lives in the main thread, not the worker. The worker is stateless — it receives a payload (code + corpus chunk), runs once, posts results back, and exits. Main thread manages all scheduling.
